# Project Context

- **Owner:** Emanuel Palm
- **Project:** gEcho — VS Code extension for recording, replaying, and generating reproducible GIFs from VS Code interactions
- **Stack:** TypeScript, VS Code Extension API (1.101.0+), Node.js, ffmpeg, JSON workbook format (.gecho.json)
- **Created:** 2026-04-08

## Key Platform Details

- **macOS:** AppleScript for window bounds, `avfoundation` capture input
- **Linux:** `xdotool`/`xwininfo` for window bounds, `x11grab` capture (Wayland not supported)
- **Windows:** PowerShell for window bounds, `gdigrab` capture
- **Replay engine:** Steps executed sequentially via VS Code command API

## Learnings

- **macOS Automation permission — CGDisplayBounds vs NSScreen (2026-04-10):** `osascript` with `use framework "AppKit"` triggers Apple Events routing, which causes macOS to show the Automation permission dialog ("would like to access data from other apps"). `swift -e` with CoreGraphics `CGDisplayBounds` is a direct framework call — no Apple Events, no permission prompt. For any macOS screen/display query that doesn't need window titles or app state, prefer CoreGraphics over AppKit-via-osascript. Inline coordinates into the Swift source string rather than using `CommandLine.arguments` with `swift -e` (arguments after the `-e` flag are not forwarded to the script).

- **Issue #2 audit (2026-04-08):** Fully implemented prior to audit. All 8 step types live in `src/types/workbook.ts`. `validateWorkbook()` in `src/workbook/workbook.ts` (via `isValidStep()`) rejects unknown step types and steps missing required fields — satisfies the validator completeness note. JSON Schema covers all 8 types; example workbook and `contributes.jsonValidation` are in place. Build clean. Issue closed.

- **TypeStep.delay semantics**: The recorded `delay` field stores elapsed ms since `startTime` (a timestamp), not a per-character typing interval. During replay, `step.delay` is treated as the per-char delay between keystrokes. These are different semantics — manually authored workbooks use `delay` as typing cadence (e.g. 55 ms/char), while recorder-generated workbooks use it as a recording timestamp.
- **ffmpeg SIGINT vs SIGKILL**: ffmpeg must receive SIGINT (not SIGKILL) to flush its output buffers and finalize the file. On some platforms, ffmpeg exits with code 255 when killed by SIGINT — treat both 0 and 255 as success in `stop()`.
- **ffmpeg startup detection**: ffmpeg writes to stderr immediately when it begins encoding. Resolving the `start()` promise on first stderr data (rather than a fixed timeout) is a reliable event-driven signal that the capture has actually started.
- **Node16 module resolution**: All imports of local `.ts` files require the `.js` extension in import paths (compiled output uses `.js`). This is enforced by `moduleResolution: Node16` in tsconfig.
- **CommandStep.args is `unknown`**: The type is `unknown`, not `unknown[]`. During replay, the args must be narrowed: spread if array, pass as single arg otherwise.
- **Ralph routing word-boundary fix (2026-04-09):** Keyword routing uses word-boundary regex (`\b${keyword}\b`) to avoid false-positives (e.g. "perspective" matching "spec"). Added `matchesKeyword(text, keyword)` helper. Routing table includes Epoch (Backend Dev) with keywords: `['api', 'backend', 'server', 'database', 'endpoint', 'epoch']`.
- **ffmpeg early-exit race condition (2026-04-08)**: When ffmpeg fails immediately after printing its version header (e.g. avfoundation permission denied, wrong device index), the `start()` promise resolves (first stderr data received) but then the close handler fires, setting `this.ffmpegProcess = null`. When `stop()` is later called and sees `proc` is null, it must verify the output file actually exists before returning — otherwise the GIF converter gets a silent non-existent file path and fails with a cryptic "No such file or directory" error.
- **ffmpeg exit code 254 on macOS**: On some macOS/avfoundation combinations, ffmpeg exits with code 254 (= -2 in signed 8-bit = SIGINT signal 2) when stopped via SIGINT. Accept 0, 254, and 255 as success codes in `stop()`.
- **GIF recording state must be 'recording-gif'**: `startGifRecording` must set `currentState = 'recording-gif'`, not `'recording'`. The status bar uses the state to render context-appropriate UI (label + click command). Using the wrong state breaks status bar interactivity during GIF recording.
- **Split stop/convert error handling**: `stopGifRecording` should use two separate try/catch blocks — one for `capture.stop()` (reports "Failed to stop GIF recording") and one for the save dialog + conversion (reports "GIF conversion failed"). A single catch merges two distinct failure modes into one unhelpful message.

- **waitForReady pattern (2026-04-08)**: `start()` resolves on first stderr line (ffmpeg version banner), which fires BEFORE avfoundation attempts to open the device. Permission denials and missing devices cause ffmpeg to exit ~200–500ms later — after `start()` has already resolved. The fix is `waitForReady(800ms)`: attach a `once('close')` listener on `this.ffmpegProcess`; if the process exits within the window, reject immediately with the stderr tail; if the timer fires and the process is still alive, remove the listener and resolve. This is purely event-driven — the timer is only a deadline, not a sleep. `setState('recording-gif')` must be moved to AFTER `waitForReady()` succeeds, so the status bar never enters recording state for a phantom capture.
- **Platform-aware error hints**: On macOS, permission errors from avfoundation show up as "AVFoundation" in stderr. After a `waitForReady` rejection, check `process.platform === 'darwin'` + `/permission|AVFoundation/i.test(msg)` and append a specific System Settings path to the error message. This applies to both `startGifRecording` and `startReplayGifRecording`.
- **Proactive Screen Recording permission check (PR #66)**: Added `checkScreenRecordingPermission(ffmpegPath?)` in `src/screen/capture.ts`. AVFoundation only lists "Capture screen" devices when the permission is active — an empty device list is the reliable signal. Extracted `enumerateAvfDevices()` to share the single ffmpeg `-list_devices` invocation between the permission check and `getScreenCaptureDeviceIndex` (ffmpeg runs at most once, result cached in `cachedAvfDevices`). `ScreenCapture.start()` calls the check at the top of the darwin branch and throws a clear error before any capture attempt. `extension.ts` calls the check on activation (darwin only) and surfaces a `showErrorMessage` with an 'Open System Settings' action button that deep-links to `x-apple.systempreferences:com.apple.preference.security?Privacy_ScreenCapture`.
- **enumerateAvfDevices Promise-cache pattern (PR #66 review)**: Replaced `cachedAvfDevices: string[] | null` with `avfDeviceEnumeration: Promise<string[]> | null`. Caching the Promise (not the resolved value) ensures concurrent callers share the same in-flight ffmpeg invocation and no duplicate processes are spawned. ENOENT/EACCES spawn errors are explicitly rejected with a `gEcho:` prefix message and the cache is cleared so callers can retry after fixing the ffmpeg path. ffmpeg non-zero exit from `-list_devices -i ""` is expected and not treated as an error — only the `err.code` check distinguishes real spawn failures.
- **User-facing error prefix consistency**: All errors thrown from `src/screen/capture.ts` must begin with `gEcho:`. The permission-not-granted error in `ScreenCapture.start()` was missing the prefix — fixed in PR #66 review.
- **swift -e is never fast (PR #67 follow-up)**: `swift -e {code}` JIT-compiles the inline snippet on every invocation — there is no caching. On first invocation this takes 10–30 s; combined with the 5 s `execFile` timeout it means `getWindowDisplayIndex()` always timed out and returned 0, while still blocking `start()` for 5 s. The fix: compile the snippet once to `~/.cache/gecho/display-index` with `swiftc -O`. The resulting native binary runs in ~1 ms and persists across VS Code sessions. Module-level `compiledBinaryPath: Promise<string> | null` caches the in-flight compilation so concurrent callers share one `swiftc` invocation.
- **stop/start race condition in ScreenCapture**: `stop()` checks `this.ffmpegProcess` which is null while `start()` is awaiting any slow async work (like `getWindowDisplayIndex()`). If the user clicks Stop before ffmpeg spawns, `stop()` sees null, the output file doesn't exist yet, and throws a confusing error. Fix: added `_startPromise` (tracks in-flight `_doStart` Promise) and `_stopRequested` flag. `stop()` awaits `_startPromise` before checking ffmpegProcess; `_doStart()` checks `_stopRequested` after `getWindowDisplayIndex()` and returns early (without spawning) if cancellation was requested.

- **PR #67 review fixes (2026-04-10):** Fixed all CI lint errors and review comments. Key patterns: (1) TypeScript `prefer-const` on timer variables — move `let` declarations to `const` inline after the `proc.on('close')` registration; closures capture by reference, so the const is initialized before the async handler fires. (2) Unused parameters in event callbacks: TypeScript event emitters accept callbacks with fewer parameters than the event signature, so `(_chunk: Buffer) => {}` can safely become `() => {}`. (3) When adding a cancellation guard after an async call, check `_stopRequested` immediately after the await — device enumeration (AVFoundation) can take 2–5 s and the user may have called `stop()` during that window. (4) `JSON.parse()` output from external binaries needs shape validation before trusting the fields — always check key field types and fall back to FALLBACK_INFO if the structure is wrong. (5) All user-facing error messages in capture.ts should have the `gEcho:` prefix for consistent VS Code notification display.
- **edit tool and em dashes:** The `edit` tool sometimes fails to match strings containing em dashes (`—`, U+2014) or backtick template literals with embedded single-quoted strings. When edit fails silently (reports no match), use Python `str.replace()` via bash to patch the file directly — it handles all Unicode correctly and is reliable for multi-line replacements.

- **PR #67 deep review — stop() error handling and $DISPLAY parsing (2026-04-10):** Reviewed recording engine changes in `fix/macos-display-index-no-automation-permission`. Key findings: (1) stop() error surfacing is CORRECT by design — `if (startErr)` throws unconditionally, prioritizing real errors (permission denied, bad ffmpeg path) over clean cancellation messages. The cancellation-only path requires _doStart() to check _stopRequested and return early (lines 135, 141). This is intentional per the comment "surface that error even if stop() was called." (2) AVFoundation args order is correct: all input options (`-f avfoundation`, `-framerate`, `-pixel_format`) come before `-i`, output options (`-vf`, `-vcodec`) after. (3) Retina math in cropFilter.ts is correct: scales crop coordinates to physical pixels (`bounds × scaleFactor`), then scales output back to logical size. Works correctly when scaleFactor = 1 (non-Retina). (4) $DISPLAY parsing handles all valid formats (`:0`, `:0.0`, `localhost:10.0`) correctly. Fixed edge case where `DISPLAY=:` (malformed, no number) would produce empty string — changed `?? '0'` to `|| '0'` to catch empty string from split(). (5) Swift binary JSON output matches platform.ts expectations exactly; CGPreflightScreenCaptureAccess() is correct API for Ventura/Sonoma and does not trigger prompts. (6) Platform cache behavior is correct: clearWindowInfoCache() called at start of every recording, so display changes between recordings are handled. No issues found with AVFoundation/x11grab/gdigrab capture args or filter syntax. Compile and lint pass. **Pattern:** When reviewing error-surfacing logic in async/cancellation scenarios, trace ALL code paths (error before cancel, cancel before error, concurrent) and verify the condition checks use the RIGHT snapshot of flags (before or after mutation). For malformed input validation, test empty string separately from undefined/null — `?? default` doesn't catch `''` but `|| default` does.
